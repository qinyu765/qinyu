出门带笔记本这件事，从我换了天选四之后就变得越来越痛苦。机器本身两公斤出头，加上那个硕大的电源适配器，背包塞满之后肩膀先投降。问题是我日常开发依赖的环境、项目、GPU 全在这台 Windows 上，换一台轻薄本意味着从零搭一遍——光是把 Node、Python、Docker、各种 CLI 工具配齐就要大半天，更别提显卡算力没法带走。

后来思路变了：既然算力搬不走，那就把屏幕搬过来。入了一台 MacBook Air M5，只负责显示和输入，所有重活还是天选四在家里扛。听起来像是被说烂了的远程桌面方案，但真正要用起来才会发现，"远程桌面"这四个字背后至少藏着三条彼此独立的技术路径，而且它们的适用场景几乎不重叠。

## 两台机器之间需要几条路

理清需求之后，我发现日常场景可以切成三块。写代码、看文档、处理日常办公，这些操作的共同特征是画面变化少、需要剪贴板互通，适合用微软自家的 RDP（Remote Desktop Protocol）。偶尔要启动游戏跑个挂机脚本，画面是全屏 3D 渲染，RDP 在这种场景下画质会断崖式下降，所以游戏串流交给 Sunshine + Moonlight。还有一条路留给 AI 编码助手——Antigravity 需要 SSH 连进一个 Linux 环境，Windows 原生不支持，于是 WSL 成了中间层。

三条路各司其职，互不干扰。但把它们串起来的网络层只有一个：Tailscale。不管 Mac 在家连 Wi-Fi 还是出门用手机热点，Tailscale 给每台设备分配的 IP 地址绑定的是设备身份而非物理网络，所以 `100.94.230.53` 这个地址在任何网络环境下都指向我的天选四，免去了动态 IP 和端口映射的麻烦。

## RDP 和 Sunshine 到底有什么不同

表面上看都是"把远程画面传过来"，但 RDP 和 Sunshine 的工作原理从根子上就走了两条完全不同的路。

RDP 的设计哲学是"描述变化"。Windows 的图形子系统在绘制每一帧时，RDP 服务会拦截所有绘图调用——画矩形、渲染文字、移动窗口——然后把屏幕划分成网格，对比前后两帧，只把发生变化的区域（微软称之为"脏矩形"）压缩后发送给客户端。客户端收到这些差异指令后在本地重新绘制。这意味着当我在 VS Code 里写代码时，画面大部分区域是静止的，只有光标闪烁的那个 2x16 像素的小块在变化，传输数据量可能只有几十字节。拿来看文档、写代码，带宽几乎不消耗。

但一旦画面开始大面积运动——比如拖动窗口、滚动网页、播放视频——整个屏幕都变成了"脏区域"，RDP 就得每帧都传完整的压缩位图。偏偏它的编码全靠 CPU 软编码（RemoteFX 编解码器），CPU 又同时在跑系统进程和应用程序，编码资源争抢之下丢帧和模糊就成了常态。

Sunshine 走的是完全相反的路线：把桌面当作一路实时视频来直播。它通过 DXGI Desktop Duplication API 直接从 GPU 的帧缓冲区抓取完整画面，然后交给 RTX 4060 上的 NVENC 专用编码单元编成 H.265 视频流。NVENC 是显卡芯片内部一块独立于 CUDA 核心的硬件电路，专门干视频编码这一件事，跟 3D 渲染管线完全不抢资源。编码后的视频帧通过 UDP 发送，Mac 的 M5 芯片用自带的硬件解码器直接解码显示。整条管线从捕获到编码到打包，延迟大约 3-5 毫秒，远低于 RDP 的 10-30 毫秒。

代价也很明显：Sunshine 不管画面有没有变化，每一帧都会完整编码并发送。5 Mbps 码率下每小时吃掉 2.25 GB 流量，用移动热点的时候需要格外留意。

| | RDP | Sunshine |
|--|-----|----------|
| 传输内容 | 绘图指令 + 脏矩形位图 | 完整 H.265 视频流 |
| 编码方式 | CPU 软编码，只传变化区域 | GPU 硬件编码，每帧完整编码 |
| 编码延迟 | ~10-30ms | ~2-5ms |
| 静态画面流量 | 极低 | 持续消耗 |
| 动态画面质量 | 明显劣化 | 接近本地体验 |
| 剪贴板 | 双向支持 | 不支持 |

## 丢一个包会怎样

编码方式决定了画质上限，但网络协议决定了稳定性下限。RDP 跑在 TCP 上，TCP 的核心承诺是每个字节都必须送达。一旦出现丢包，发送端会停下来等确认、超时、重传，接收端才能继续处理后续数据。弱网环境下一个丢包可能引发 50-200 毫秒的停顿，连续丢包还会触发 TCP 拥塞控制主动降速，画面直接冻住。

Sunshine 用的是 UDP 加前向纠错（FEC）。UDP 不保证送达——丢了就丢了，但 FEC 会随每个数据包附带冗余信息，丢失少量包时可以从冗余中恢复。丢包严重时画面会出现短暂的马赛克色块，下一帧立刻恢复，不会冻屏。体验有点像微信视频通话：偶尔糊一下，但不会卡住不动。在 3% 丢包率下，RDP 可能画面冻结然后突然跳帧，Sunshine 只是某一帧出现色块、下一帧就恢复正常，人眼几乎感知不到。

## Tailscale 把两台机器拉进同一个局域网

不管用 RDP 还是 Sunshine，首先要解决的问题是：Mac 怎么找到家里的 Windows。传统 VPN 走星型拓扑，所有流量经过中心服务器中转，延迟等于两段之和。Tailscale 走的是 Mesh 网状拓扑，底层用 WireGuard 加密隧道，两台设备之间直接通信。

连接建立的过程大致是这样的：两台设备各自向 Tailscale 的云端协调服务器注册，上报公钥、内网 IP、公网 IP 和 NAT 类型。协调服务器把对方的信息下发给两端后，双方同时向对方的公网地址发 UDP 包尝试 NAT 打洞。如果 NAT 类型允许（Cone NAT），打洞成功就建立直连，延迟可以低到 1-5 毫秒；如果是对称型 NAT 打洞失败，退回 DERP 中继服务器，延迟会上升到 50-200 毫秒。

WireGuard 之所以适合串流场景，是因为它的加密开销几乎可以忽略。ChaCha20-Poly1305 对称加密在现代 CPU 上有原生指令集支持，内核态实现避免了用户态切换开销，单核就能处理超过 1 Gbps 的加密吞吐，额外延迟不到 1 毫秒。

有一个容易踩的坑：如果两台机器都开着 Clash Verge 的 TUN 模式做代理，Tailscale 的流量可能被代理拦截后绕路出去。我的配置里专门加了几条 DIRECT 规则，把 Tailscale 的控制面板域名、进程名和 `100.64.0.0/10` 子网全部设为直连。漏了这些规则的后果是 `tailscale netcheck` 显示最近的 DERP 节点在洛杉矶——延迟两百多毫秒，串流直接没法用。

## 场景分工：谁干什么活

跑了几个月下来，两套方案的分工已经很清晰了。日常开发和办公全走 RDP，流量低、剪贴板互通、多显示器支持也好。需要启动游戏或者跑自动化脚本的时候切到 Moonlight，短暂连接把游戏拉起来，确认脚本开始运行之后就断开——反正脚本跑在 Windows 本地，不需要持续串流。后面想看看脚本状态，用 RDP 轻量连进去瞄一眼就行，比持续开着 Moonlight 省太多流量。

SSH 那条路稍微特殊一些。我用的 AI 编码助手 Antigravity 支持通过 SSH Remote 连接远程机器进行开发，但它要求目标环境是 Linux——Windows 原生的 OpenSSH Server 不满足条件。所以我在天选四上装了 WSL（Windows Subsystem for Linux），跑了一个 Ubuntu 22.04，专门为 IDE 提供 Linux 环境。SSH 本身是一套基于公钥密码学的安全远程登录协议：客户端和服务端通过非对称密钥完成身份验证，之后所有通信走对称加密通道。我在 Mac 端用 `ssh-copy-id` 把公钥部署到 WSL，之后连接不再需要输密码——`ssh win-wsl` 一条命令直达。WSL 的 SSH 端口设成了 2222，避免跟 Windows 自带的 22 端口打架。

WSL 的网络配置选了 Mirror 模式，这个模式让 WSL 直接共享宿主机的整个网络栈，包括 Tailscale 分配的地址。这样 Mac 通过 Tailscale IP 就能直接 SSH 进 WSL，不需要额外做端口转发。项目文件通过 `/mnt/d/` 挂载点访问 Windows 磁盘，再用软链接映射到 home 目录下，编辑的是同一份文件。跨文件系统的 IO 性能确实比 WSL 原生差一截，`npm install` 之类的密集文件操作会慢些，但日常写代码、跑测试完全够用。

回头看，这套方案的核心思路就一句话：把计算留在性能最强的机器上，只在终端之间传输画面和输入。Mac 负责提供一块好屏幕、一个舒服的键盘和一整天的续航，天选四负责所有的计算、编译和 GPU 运算。两公斤的机器和硕大的电源适配器留在家里吃灰，出门只带一台一公斤出头的 Air，反而什么都没少。
